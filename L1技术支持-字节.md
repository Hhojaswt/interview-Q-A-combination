- ### 1.  如何设置防火墙：

**Linux: 使用 iptables (通用性强)**

查看规则：iptables -L -n

允许 SSH (22端口)：iptables -A INPUT -p tcp --dport 22 -j ACCEPT

允许 Web 服务 (80/443)：iptables -A INPUT -p tcp --dport 80 -j ACCEPT

拒绝所有其他入站：iptables -A INPUT -j REJECT

注意：配置完规则后务必保存（如 service iptables save），否则重启后失效。

**使用 UFW (Ubuntu 推荐)**

启用：sudo ufw enable

允许端口：sudo ufw allow 22 (允许 SSH) 或 sudo ufw allow 80/tcp

拒绝连接：ufw deny 23         # 拒绝Telnet

**使用 firewall**
firewall-cmd --state
firewall-cmd --list-all

停止服务
systemctl stop firewalld
systemctl disable firewalld

- ### 2.  对象存储是什么 对象存储和文件的关系是什么 有哪些桶设置的方式

对象存储是一种用于管理海量非结构化数据的现代数据存储架构。

它与传统的文件存储方式有本质区别。简单来说，它*不再使用文件夹和子文件夹的层级结构来组织数据*，而是将所有数据（如照片、视频、音频、文档等）都视为一个个独立的“对象”（Object），并将它们扁平化地存放在一个巨大的“桶”（Buckket)中

**🗂️ 对象存储与文件的关系**

你可以将对象存储中的概念与文件存储进行类比，以便更好地理解：
对象 (Object)：相当于文件。它是存储的基本单元，不仅包含文件数据本身，还附带了丰富的元数据（如创建时间、内容类型、自定义标签等）和一个全局唯一的标识符。

桶 (Bucket)：相当于一个顶级容器。你可以把它想象成一个根目录，但它内部是扁平的，没有多层级的文件夹结构。所有的对象都直接存放在桶里。

**1. 基础 ACL（访问控制列表）**
这是最简单粗暴的。直接在桶的 ACL 设置里，把“公共访问”或者“所有人（Everyone/All Users）”的权限勾选为 Read（读取）。

效果：只要知道文件的 URL，谁都能下载。

**2. Bucket Policy（桶策略）—— 最推荐，最专业**
这种方式最灵活。它是用一段 JSON 脚本来定义的。比如我想让 static 目录下所有的图片都能被公网访问，我会写个 Policy：

Effect: Allow

Principal: *（代表所有人）

Action: s3:GetObject

Resource: arn:aws:s3:::my-bucket/*

**3. 静态网站托管（Static Website Hosting）**
如果用户是为了刷网页用的，开启这个功能后，桶会自动变成一个 HTTP 服务器。

特点：它会强制要求桶里的内容具备公共读权限，否则网页打不开。

- ### 3.  service之间连接出现问题怎么排查

**🎯 第一步：检查 Service 与 Endpoints**

Service 是 Pod 的抽象和访问入口，首先要确认 Service 本身配置正确，并且有健康的后端 Pod。

- 检查 Service 配置

使用 kubectl describe svc <service-name> 命令，重点关注以下几点：
- 类型 (Type)：确认是 ClusterIP、NodePort 还是 LoadBalancer。ClusterIP 只能在集群内部访问。
- 端口 (Ports)：检查 port（服务暴露的端口）和 targetPort（容器监听的端口）是否配置正确，确保流量能被正确转发到容器的应用端口。
- 选择器 (Selector)：确认 Selector 的标签（Labels）是否与目标 Pod 的标签完全匹配。

- 检查 Endpoints

Endpoints 是 Service 背后所有健康 Pod IP 的列表。这是排查中最关键的一步。

使用 kubectl get endpoints <service-name> 命令。

如果 ENDPOINTS 为空或显示 <none>，说明 Service 没有关联到任何 Pod。这通常由以下原因导致：
- 标签不匹配：Service 的 selector 与 Pod 的 labels 不一致。
- Pod 未就绪：Pod 虽然处于 Running 状态，但其就绪探针 (Readiness Probe) 检查失败，导致 Kubernetes 不会将其 IP 加入 Endpoints。
- 没有运行中的 Pod：对应的 Deployment 副本数为 0 或 Pod 启动失败。

**🌐 第二步：测试连通性与 DNS**

确认 Service 有后端 Pod 后，需要测试网络层面的连通性和服务发现是否正常。

1. 创建调试 Pod
为了在一个干净的环境中测试，可以创建一个包含常用网络工具的临时 Pod：

kubectl run -it --rm debug-pod --image=curlimages/curl --restart=Never -- sh

2. 测试 DNS 解析
Service 之间通常通过服务名进行访问，这依赖于集群内部的 DNS（CoreDNS）。
- 在调试 Pod 中执行 nslookup <service-name>。
- 如果解析失败，问题可能出在 CoreDNS 服务上。可以检查 kube-system 命名空间下的 coredns Pod 是否正常运行，并查看其日志。
- 尝试使用完全限定域名（FQDN）进行解析，例如 <service-name>.<namespace>.svc.cluster.local，以排除命名空间问题。

3. 测试网络连通性
如果 DNS 解析正常，接下来测试实际的 HTTP/TCP 连接。
- 在调试 Pod 中执行 curl -v http://<service-name>:<port>。
- 连接被拒绝 (Connection refused)：通常意味着流量到达了目标 Pod，但目标应用没有在指定端口上监听。检查应用的启动日志和容器端口配置。
- 连接超时 (Connection timed out)：通常意味着请求被防火墙或网络策略拦截，或者网络路由本身存在问题。

**🛡️ 第三步：审查网络策略**

网络策略（NetworkPolicy）是 Kubernetes 中的“防火墙”，可能会阻止 Service 之间的流量。
- 使用 kubectl get networkpolicies 检查当前命名空间下是否存在网络策略。
- 使用 kubectl describe networkpolicy <policy-name> 查看具体规则，确认策略是否允许了源 Service 的 Pod 访问目标 Service 的 Pod。
- 一个常见的错误是创建了默认的“拒绝所有入站流量”的策略，导致所有外部访问都被阻断。

**⚙️ 第四步：排查底层网络组件**

如果以上步骤均未发现问题，则需要深入检查集群的底层网络组件。

1. 检查 kube-proxy
kube-proxy 负责在节点上维护网络规则（如 iptables），以实现 Service 的负载均衡。

2. 检查 kube-system 命名空间下的 kube-proxy Pod 是否正常运行。
查看其日志，寻找如“iptables sync failed”之类的错误信息。

3. 检查 CNI 插件
容器网络接口（CNI）插件（如 Calico, Flannel）负责 Pod 之间的网络连通。
- 检查 CNI 插件在 kube-system 命名空间下的 Pod 状态。
- 查看 CNI 插件的日志，排查是否有网络配置或路由相关的错误。

- ### 5.  用过哪些工具：

**🛠️ 核心命令行工具 (CLI)**
| 工具名称 | 核心用途 | 为什么它是神器？ |
| :--- | :--- | :--- |
| kubectl | 基础控制 | 万能钥匙。特别是 `kubectl get/describe/logs/exec` 是必修课。 |
| kubectl debug | 临时容器注入 | 强烈推荐。当你的业务镜像（如 Alpine/Scratch）没有 `ping`、`curl` 时，它能注入一个包含所有工具的调试容器到目标 Pod 的命名空间中。 |
| Stern | 日志聚合 | 原生的 `kubectl logs` 只能看一个 Pod。Stern 支持多 Pod 同时日志流（例如 `stern -l app=myservice`），还能给不同容器标上颜色，排查微服务调用链时非常直观。 |
| Netshoot | 网络诊断镜像 | 一个集成了 `tcpdump`, `dig`, `nslookup`, `mtr`, `netstat` 等 70+ 网络工具的镜像。通常配合 `kubectl debug` 使用，或者作为一个独立的调试 Pod 运行。 |
| Telepresence | 本地调试 | 允许你在本地机器运行代码，但流量代理到远程集群。当你想调试本地代码如何与集群内的数据库交互时，它能省去反复部署的麻烦。 |
