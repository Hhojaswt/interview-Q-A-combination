** 1. POC（Proof of Concept）
概念：概念验证，是项目初期用来“验证功能可行性”的测试小环境；

目的：客户用它评估产品能力 / 易用性 / 性能 / 接口适配性；

示例：比如构建一套简化版的数据采集 → 存储 → 可视化流程，测试 Azure Monitor + Log Analytics + Grafana 是否满足告警需求。

** 2. License / Instance 定价方式
Instance（实例）：代表虚拟资源（VM、数据库、容器等），按小时或月计费；

License（许可）：代表软件授权，如 SQL Server Enterprise，每核需购买 License；

计费模式：

Pay-as-you-go（按量付费）；

Reserved（预留实例价格优惠）；

Hybrid Use Benefit（已有本地 License 可抵扣云端 License）；
示例：客户部署 SQL Server VM 时，Azure 会额外计入 SQL License 成本。

## 产品定价逻辑
✅ 一、VM（虚拟机）定价逻辑
- 核心公式：
VM 总价 = 实例费用（Instance）+ 软件 License（如 OS、SQL）+ 存储（OS Disk / 数据盘）+ 网络带宽（出站）

1. 实例规格计费（Instance Price）
计费维度：vCPU 数量、内存大小、使用时间（按小时或秒）

常见系列：

B 系列（经济型，适合低负载）

D/E 系列（通用型，适合 Web 应用）

F/H 系列（高性能计算）

L 系列（IO 密集型）

计费方式：

Pay-as-you-go：按使用时长付费，适合短期项目

Reserved Instance（RI）：预付费 1~3 年，享受大额折扣

Spot VM：价格浮动，适合容错任务

2. License（软件授权）
Windows / SQL Server 等操作系统或软件需要额外收费

支持：

按小时计费 License（如 Windows Server）

Hybrid Use Benefit（客户已有 License 可抵扣）

3. 磁盘（Storage）
OS Disk + Data Disk

分为 Premium SSD、Standard SSD、HDD

按 容量（GiB） + 类型 + IOPS 定价

4. 网络（出站带宽）
入站（Inbound）免费

出站流量按 GB 收费

Azure 中国区：0-10TB 起步价 0.61 RMB/GB，逐级递减

| 场景      | VM 系列  | 购买方式           | 磁盘建议                |
| ------- | ------ | -------------- | ------------------- |
| 开发测试    | B 系列   | 按需/Spot        | 标准 HDD              |
| 通用生产业务  | D 系列   | 保留实例           | Premium SSD（P20+）   |
| 大内存数据库  | E/M 系列 | 保留实例 + Hybrid  | Premium SSD（P30+）   |
| 高并发 API | F 系列   | 按需 + AutoScale | Ultra Disk（如需超高 IO） |
| 临时批量作业  | F/L 系列 | Spot           | 临时磁盘即可              |
| 容器集群节点  | D/E 系列 | 按需 / 保留        | Standard SSD（节省成本）  |


✅ 二、数据库定价逻辑（以 Azure SQL 为例）
模式一：DTU 模型（老旧）
DTU（Database Transaction Unit）：代表 CPU、内存、IO 的一个打包计价单位

单一价格，缺乏弹性，已逐步被弃用

模式二：vCore 模型（主流）
定价维度：

vCore 数量

存储容量（GB）

备份存储（超过7天要额外收费）

购买方式：

普通：按需计费

Reserved capacity：1~3 年预付费折扣

License：

Azure Hybrid Benefit 可使用已有 SQL License 减免

模式三：Serverless
自动伸缩计算资源，适合间歇性访问的数据库

按 vCore 秒数 + 存储计费

✅ 三、网络服务定价逻辑
1. 公网带宽
出站流量按 GB 收费（Ingress 免费）

不同区域价格不同，跨区域传输也按量收费

2. Load Balancer
Basic Load Balancer：免费

Standard Load Balancer：

固定费用 + 规则数 + 处理流量量级

3. Application Gateway / Front Door / CDN
按如下维度计费：

实例数量

规则数量

数据处理量（GB）

WAF 启用情况

4. ExpressRoute（专线）
两种模式：

Unlimited Data Plan（包月）

Metered Data Plan（按量计费）

根据带宽等级（50 Mbps ~ 10 Gbps）和地理区域计价


✅ 1. 你如何识别客户的真实业务痛点？
答题框架（STAR法）：

S（场景）：客户往往带着技术问题来找我们，比如“想做监控”“要迁云”，但很多时候只是表象。

T（任务）：我的任务是通过深入沟通，将这些“技术目标”转化为“业务目标”。

A（行动）：

先通过访谈了解他们的业务流程、组织架构、关键指标；

比如我曾接触一个教育行业客户，他们希望对线上课程平台进行性能优化，但深入后发现，问题核心在于用户高峰期时段接口响应慢导致掉线，影响转化率。

我将痛点总结为：“核心系统在峰值时段缺乏弹性扩缩容能力，缺失实时监控预警机制”。

R（结果）：我们提出了基于 AKS + HPA 的方案，结合 AMP 做实时接口告警，帮助客户将高峰期稳定率提升了约 35%。

关键词：调研、流程梳理、业务场景还原、非技术语言转技术方案

✅ 2. 遇到客户提出不合理或成本高的需求时你怎么应对？
答题建议：

首先尊重客户的表达，不直接否定；

评估目标本质：客户想做多活，但预算只有单活的成本 → 是高可用性需求，而不是“非得异地双活”；

提出更具性价比的替代方案：比如用 Zone-Redundant Storage + Read Replicas + 自动备份，而不是部署两套生产环境；

量化对比：“双活投入需要多出 2 倍资源和运维复杂度，我们建议使用 XX 架构，投入下降 40%，并保留 99.95% SLA。”

关键词：换位思考、目标导向、成本对比、技术影响可视化

✅ 4. 你设计的方案中如何保证高可用和数据安全？
答题建议：

高可用方面：

计算资源使用 AZ / Region 分布（VMSS、AKS 多节点池）；

数据使用 GRS 存储、数据库副本（Active Geo-Replication）；

应用层接入层采用 Azure Front Door + Load Balancer；

数据安全方面：

全面使用 RBAC + Entra ID + Key Vault 管理凭据；

数据传输加密（HTTPS / TLS 1.2），静态加密（Storage 加密、TDE）；

备份与恢复机制：使用 Azure Backup / Snapshot / 自动快照保留；

关键词：高可用（冗余 + 容灾）、数据安全（加密 + 授权 + 备份）

✅ 5. 项目中你如何与售后交付团队进行配合，避免方案与交付脱节？
答题建议：

项目初期即邀请交付团队参与需求评审；

所有方案设计通过交付可落地评审（包括资源类型、账号权限、安全需求等）；

输出可交付物清单（部署图、模块说明、接口协议、监控指标）；

落地时提供 PoC Demo 环境供交付侧验证；

交付后参与客户 UAT / 试运行阶段，做性能/故障验证；

关键词：提前介入、可交付物完整性、交付协同机制、角色清晰

✅ 高可用的涵盖层面：
1️⃣ 计算资源高可用（VM / 容器 / App）
虚拟机层面：

使用 Availability Set（不同故障域/更新域） 或 Availability Zone（物理隔离区域）部署 VM。

建议启用 VM Scale Set（VMSS）+负载均衡器 实现弹性扩缩容。

避免将单点服务部署在单台 VM 上。

容器服务（如 AKS）：

使用多个 Node Pool（至少3个节点、跨 Zone）；

配置 HPA（Horizontal Pod Autoscaler） 自动应对流量波动；

控制平面启用 高可用控制节点（内建 for AKS）。

2️⃣ 存储与数据高可用
Blob 存储：

使用 GRS（Geo-Redundant Storage） 保障区域级别容灾；

ZRS（Zone-Redundant） 可在 Region 内自动副本冗余。

数据库层面：

Azure SQL：启用 自动故障转移组（Auto-failover Group）；

MySQL / PostgreSQL：使用 HA 模式 + 自动备份/只读副本；

Cosmos DB：启用多区域复制（multi-master 或单写多读）；

3️⃣ 网络层高可用
入口流量：

使用 Azure Front Door 或 Application Gateway 做多区域流量调度与 WAF 防护；

使用 Load Balancer（L4）+ NSG + UDR 分发内部流量；

出站路径：

出口路径建议设置 NAT Gateway / proxy 双冗余通道；

DNS 解析使用 Private DNS + 条件转发器，保障能解析 Azure 核心域名。

4️⃣ 应用服务高可用
应用层配置：

支持无状态部署（如 Web App 使用 App Service Plan + Slot）；

微服务具备弹性：服务发现 + 自动注册/注销 + 故障恢复；

接口调用链路设置重试机制 + Circuit Breaker（断路器）防雪崩。

5️⃣ 监控与自动恢复机制
配置 Azure Monitor / Log Analytics + Alert + Action Group：

异常立即告警（如 CPU飙升 / Pod重启）；

结合 Azure Automation 或 Logic App 自动执行恢复动作；

例如检测到数据库 CPU > 80%，自动 scale up 或重启节点。

✅ 1. Availability Set（可用性集合）
🎯 定义：
将一组虚拟机放入一个 Availability Set，可以确保它们分布在 不同的故障域（Fault Domain） 和 更新域（Update Domain） 中，从而降低硬件故障或计划性维护导致的整体宕机风险。

📦 组成概念：
Fault Domain（FD）：

表示 Azure 数据中心中 物理硬件隔离 的单元。

不同 FD 代表不同的机架、不同的电源、网络交换机等。

如果某个 FD 故障，其他 FD 不受影响。

Update Domain（UD）：

表示 Azure 进行 计划性维护时的逻辑批次。

Azure 不会同时对同一 Availability Set 中的所有 UD 进行更新。

✅ 2. Availability Zone（可用性区域）
🎯 定义：
Availability Zone 是 Azure 提供的 物理隔离故障域，一个 Region 通常包含 3 个或更多的 AZ，每个 AZ 都有独立的 电力、网络、冷却系统和数据中心建筑。

✅ 特点：
VM、托管磁盘、负载均衡器等资源可部署到不同的 Zone（Zone 1、Zone 2、Zone 3）。

如果某个 AZ 故障，其他 Zone 中的服务可以照常运行。

可实现跨数据中心级别的高可用。
